01

The question

We scrape 11 football data sources into one scrape.* schema on staging Supabase — ~16.5M rows today. Each source is mirrored source-faithfully (one table-set per source, native IDs, raw jsonb kept). The recurring question from anyone designing a backend or frontend on top: which source gives what, how fresh is it, and — if we scraped faster — could we get live commentary, live player stats, live xG?

Short answer up front: scraping faster only helps the two live-capable sources (ESPN, OpenLigaDB). Everything else is historical or daily-batch — polling harder won't make it fresher, because the upstream only changes daily or weekly. And on free + legally-clean sources, the live ceiling is live score + team-level stats — not goals-with-scorer, not lineups, not player-level live stats, not live xG. Those need a grey-zone API (legal risk) or a paid feed.

The one finding that surprised us

ESPN's summary endpoint — the obvious "go deeper on a live match" lever — is hollow on events. Probed live: it returns 27 team-level aggregate stats (possession, shots, fouls, corners, cards…) but no goals-with-minutes, no lineups, no per-player stats, no play-by-play, no xG. It's a stats-box, not an incident feed.

02

The four liveness classes

Every source falls into one of four classes. The class — not the scraper's poll rate — sets the freshness ceiling. Class is decided by how often the upstream changes.

Historical load once, freeze
Static or archival datasets. You ingest once; re-polling buys nothing. StatsBomb open-data, jfjelstul WC, martj42, openfootball. Depth is high (events, full results), freshness is irrelevant — the data is about the past.
Daily-batch refreshes ~daily/weekly
Upstream updates on a slow cadence. Poll once a day and you're current. Transfermarkt CC0 (weekly), Understat (~24h post-match), football-data.co.uk (weekly), ClubElo (daily). Forward-moving but never live.
Forward hours-fresh
Near-real-time-ish — minutes to hours behind. Transfermarkt news RSS (transfer news), TheSportsDB free schedule/results. Good for "what just happened", not for in-play.
Live-capable sub-minute if you poll fast
Reflects in-play state within seconds. Scraping faster does help here. ESPN, OpenLigaDB on the legal/free side; SofaScore, FotMob on the grey-zone side. This is the only class where a near-real-time poller is worth building.

07

How live can we actually get — the verdict

Faster scraping only helps two sources

Polling harder makes no source fresher unless its upstream changes that fast. Only ESPN and OpenLigaDB are live-capable. Everything else (StatsBomb, Understat, TM, fdcouk, Wikidata, the WC datasets) updates daily-to-weekly at source — a tighter poll just re-reads the same rows.

The free + legal live ceiling

Live score + 27 team stats (ESPN, global) and goal events with scorer (OpenLigaDB, Bundesliga). That's it. No lineups, no per-player live stats, no live xG, no commentary on a free + clean source.

Commentary / live player stats / live xG

Not available free + legal. The data exists on SofaScore/FotMob (shotmap xG, live player ratings, momentum) but accessing their APIs violates ToS and risks IP bans + legal exposure. For production these need a paid licensed feed (e.g. API-Football ~$19/mo, SportMonks, Opta) — excluded from this free-scope review.

Live needs a separate service

Our pipeline is daily-batch today (laptop tmux → planned GitHub Actions cron). True live can't ride that. It needs an event-driven poller: spins up when a match goes in_progress, hits ESPN/OpenLigaDB every 30–60s, writes incremental rows, shuts down at full-time. Same DB, separate process.
